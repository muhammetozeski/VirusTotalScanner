# Append a drop/rescan onto existing results instead of wiping the queue

Kategori: ux | Efor: L | Rank: 6

## Ne
ScanScheduler.RunAsync unconditionally calls UiPost(() => Items.Clear()) at line 71, so every StartScan path — drag-drop, file/folder buttons, and RescanSelected (ScanQueueControl.cs:685) — destroys all prior rows. Add ScanScheduler.AppendAsync(paths, opts) running the same expand→hash→lookup pipeline but skipping the Items.Clear() and ResetCounters wipe, instead adding new ScanItems and folding outcomes into the running counters. Scope the first cut to the not-currently-running case (append onto a finished sweep) to avoid reworking the single per-run _cts/_uploadGate/_md5Gates. In ScanQueueControl, give drag-drop and the file/folder buttons a held-Shift 'add to queue' path and make RescanSelected append the re-scanned rows rather than calling StartScan (which wipes).

## Neden
The biggest queue trap: finish a 2000-file sweep, drag in one more file to double-check, and the whole result set vanishes with no undo. RescanSelected wiping the list is an especially nasty surprise. Append-not-replace is table stakes and unlocks the natural 'scan this, then also that' flow the app currently forbids.
