# Split local vs network concurrency so local-only work scales past the fixed 2 threads

Kategori: perf | Efor: M | Rank: 11

## Ne
ScanOptions.MaxConcurrency defaults to 2 (ScanModels.cs:124) and ScanScheduler builds one ParallelOptions with that degree for the whole run (line 116). But ProcessAsync splits into a local stage (TrustService.Evaluate, HashService.ComputeAsync, cache / KnownGoodDb / allowlist checks, lines 165-205) that is CPU/disk-bound and a VT stage that is rate-limited — and the network side is ALREADY bounded independently by _uploadGate and the per-md5 _md5Gates dedup. Run the per-file pipeline at a higher degree (e.g. Environment.ProcessorCount, its own setting) and let only the actual VT network calls (DoLookupAsync) be gated by the conservative MaxConcurrency, so the local fast path is no longer throttled to 2.

## Neden
On big trees most files resolve locally (trusted-signed, cache hits, known-good, allowlist) and never touch VT, yet they crawl at 2 threads because that one knob also throttles the network. Decoupling makes an all-local re-sweep of a large folder finish several times faster on a multi-core PC with no added VirusTotal rate-limit risk.
