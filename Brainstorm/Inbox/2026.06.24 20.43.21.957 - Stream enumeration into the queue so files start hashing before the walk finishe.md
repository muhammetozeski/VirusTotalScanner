# Stream enumeration into the queue so files start hashing before the walk finishes

Kategori: perf | Efor: L | Rank: 11

## Ne
In ScanScheduler.RunAsync (lines 83-97) the scan awaits SelectionEnumerator.Expand to materialize a full List<string>, then OrderByDescending(RiskScorer.Score) over the whole list (line 92-93), then BulkAdd, all before a single file is touched - a multi-second blank-grid stall on big trees where Cancel cannot interrupt the walk (ct only checked at await). Make Expand expose a lazy IEnumerable<string> over the already-lazy Directory.EnumerateFiles, have the scheduler feed a Channel<ScanItem> that Parallel.ForEachAsync drains so hashing begins while enumeration is still discovering, and append rows in BulkAdd chunks as they arrive. Keep the current materialize-then-sort path only when RiskWeightedOrdering is on AND a pre-count is small; otherwise stream (order degrades to discovery order, surfaced to the user).

## Neden
The number-one 'app froze on a big folder' symptom is the dead time before the first row appears and an unresponsive Cancel. Progressive enumeration turns a 5-10s blank stall into instant first results and a Cancel that works immediately, which is exactly what scanning C: or a full Downloads tree feels like first.
