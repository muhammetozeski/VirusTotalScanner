# Show the parent/creator of a locking threat in the kill prompt

Kategori: reporting | Efor: S | Rank: 7

## Ne
Extend RunningProcesses.MatchingProcesses to also return each holder's parent process id and name. Get the parent via a Toolhelp32 snapshot (CreateToolhelp32Snapshot + PROCESSENTRY32.th32ParentProcessID, mapped to a name through the same snapshot) — this is lighter and needs no special access compared to NtQueryInformationProcess. Then, in TryQuarantineUnlocking (ScanQueueControl ~line 845), where the 'Bu dosya su an calisiyor: {who}' confirm prompt is already built from the holders, append 'Baslatan: <parent-name> (PID <ppid>)' per holder. When the parent can't be resolved (gone/denied), omit the line gracefully — same forgiving pattern as the existing MainModule read. No new dialog, store, or subsystem; the parent fields ride along the (Pid,Name) tuple already returned.

## Neden
The locking-process flow already names WHICH process holds the binary but never WHAT launched it — and the codebase has zero parent-process attribution anywhere (verified by grep). The parent is the most direct attribution evidence for a running threat: explorer.exe means the user double-clicked it, a browser means a drive-by, a scheduled task means persistence is firing. Surfacing it in the prompt the user is already reading lets them make the close/keep call with the launch chain in front of them, for the cost of a couple of struct fields and one P/Invoke. Unlike the other two candidates it adds no persisted subsystem and overlaps nothing on the shipped list (NeighborsService and IncidentTimelineService cover file co-arrival, not process lineage).
