# Re-validate allowlisted hashes so a later-compromised build stops being silently skipped

Kategori: protection | Efor: M | Rank: 10

## Ne
Add LastVerifiedUtc + DetectionCountAtAdd + IsStale to AllowlistEntry and an opt-in 'allowlist health' pass (new AllowlistStore method + a button in SettingsControl's allowlist card). It re-queries marked-clean hashes keyless (GuiScrapeService first, zero-quota cache otherwise, reusing RecheckService/HistoryReverdictService). If a previously-clean hash is now flagged by major engines it does NOT delete the entry; it sets IsStale=true and raises a tray toast. ScanScheduler.cs (AllowlistStore.Contains skip at ~line 207) keeps honoring the skip until the user acts; the Settings grid renders stale rows in red with a 'gözden geçir' action. Also add a one-pass 'Geçmişten temiz dosyaları içe aktar' bulk-seed from ScanHistoryStore clean entries to fill the list without one-right-click-at-a-time.

## Neden
Confirmed at the AllowlistStore.Contains skip: any allowlisted hash is suppressed forever with no re-check. If that exact build is later revealed as supply-chain-compromised and VT now flags it, the scanner silently keeps skipping it — the noise-reduction feature becomes a permanent blind spot, the one place it can turn into a security hole. Periodic re-validation keeps the convenience and closes the gap, reusing infrastructure that already exists.
